# 02 — 技術架構

## 與 GitHub / Netlify / Firebase 的關係（一句話）

程式由 **GitHub** 保存；**Netlify** 發布成 **唯一網址** 的靜態站；使用者瀏覽器載入的 `index.html` 再連到 **Firebase Realtime Database** 做多人 **即時同步**。細節見 [01-專案總覽.md](./01-專案總覽.md)。

## 整體架構

```
瀏覽器
  └── index.html
        ├── <style>  全域與元件樣式
        ├── #app     React 掛載點
        └── <script> 內嵌應用邏輯（非 JSX，使用 React.createElement）
              ├── Firebase 初始化（Realtime Database）
              ├── 常數：TEAMS、NF、RS…
              ├── 工具函式：membersFromFirebase、sortMembersByTeam…
              └── React 元件：App（內含多階段 render）
```

## 為何沒有 `src/`、`package.json`？

刻意維持 **零建置**：改完 `index.html` 即可上傳部署。代價是單檔較長、需靠註解與本 `docs/` 維持可讀性。

## React 寫法

- 使用 `const e = React.createElement`，**沒有** JSX 轉譯器。
- 條件渲染常見型態：`tab==="browse" && e("div", …)`、`cond ? a : b`。
- 狀態以 `useState` / `useEffect` / `useMemo` / `useRef` 為主。

## 應用程式階段（`phase`）

`App` 依 `phase` 分支渲染（概念上為「多個根畫面」）：

| phase | 說明 |
|-------|------|
| `loading` | 開機、嘗試自動登入 session |
| `join` | 輸入暱稱、房間碼加入 |
| `app` | 主功能（三個分頁） |
| `admin` | 管理後台（密碼頁 → 面板） |

## 主功能分頁（`tab`）

在 `phase==="app"` 時，`tab` 控制：

- `browse` — 我的選擇（卡片網格、篩選、可選只看某位房友的名單）
- `overlap` — 共同支持（聯盟／球隊／國籍篩選、計入成員勾選、顯示方式：交集／聯集）
- `members` — 房間成員列表與點擊詳情 modal

## 與 Firebase 的邊界

- **即時資料**：房間內成員選擇寫在 RTDB（見 `03-Firebase與資料.md`）。
- **啦啦隊「主名單」**：啟動時讀 `config/members`，經 `membersFromFirebase` 正規化後進入 React state（**無**程式內建名單合併；雲端為唯一資料來源）。

## 樣式

- 全寫在 `index.html` 的 `<style>`，以 class 為主（如 `.filters`、`.card`、`.og` / overlap 卡片區）。
- 無 CSS-in-JS、無外部 CSS 檔。

## 修改時建議搜尋的關鍵字

- `function App` — 主元件與 state 集中處
- `phase===`、`tab===` — 畫面分支
- `db.ref(` — 所有 RTDB 存取
- `membersFromFirebase` — 將雲端陣列轉成排序後的 `members`（無內建合併）
- `overlapFiltered` — 共同支持計算
- `ADMIN_PASSWORD` — 管理員密碼（請勿長期使用預設值）
