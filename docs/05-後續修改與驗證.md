# 05 — 後續修改與驗證

## 最常修改的位置

| 需求 | 建議位置 |
|------|----------|
| 新增／修正預設啦啦隊員 | `DEFAULT_MEMBERS` 陣列（注意 **id 唯一**、與 `TEAMS` 的 `t` 對應） |
| 新增球隊或聯盟標籤 | `TEAMS`、`TEAM_BY_LEAGUE`、`TEAM_ORDER` 衍生邏輯 |
| 國籍圖示或篩選選項 | `NF` 常數 + 各分頁 `nf` 的 `<select>` option（browse / overlap / 管理表單） |
| UI 文案或流程 | `App` 內 `return e("div", …)` 各分支 |
| Firebase 專案 | `FIREBASE_CONFIG` |
| 管理員密碼 | `ADMIN_PASSWORD`（**部署前必改**） |

## 新增國籍代碼時

除 `NF` 外，務必同步：

- 「我的選擇」「共同支持」國籍下拉選單
- 管理後台編輯成員的國籍按鈕列

否則資料有該 `nat` 但 UI 無法篩選或顯示異常。

## 語法驗證（推薦）

因主程式在內嵌 `<script>`，可用：

```bash
node scripts/check-inline.mjs
```

成功會印出 `check-inline: OK`。此腳本以 `react-dom` 之後的內嵌 `<script>` 為錨點抽出內容，再以 `vm.Script` 做語法編譯檢查。

## 與匯出 JSON 的關係

- `cheerleader-app-default-rtdb-export.json` 適合備份／還原／對照 **RTDB 實際樹狀結構**。
- 應用程式的「預設名單真相」仍以 `index.html` 的 `DEFAULT_MEMBERS` + `mergeMembersWithDefaults` 為準；匯入雲端時注意 **id 衝突與合併策略**。

## 給 AI 的修改原則（建議附在 prompt）

1. 優先 **最小修改**：能改 20 行就不要大改結構。  
2. 單檔無測試框架：改動後至少跑 `node scripts/check-inline.mjs`。  
3. 任何涉及 **規則／權限** 的問題，需同步提醒使用者檢查 Firebase Console Rules。  
4. 不要將真實 **管理密碼** 或 **敏感規則** 寫進公開 repo；文件僅描述「哪裡可設定」。

## 已知架構取捨（供後續決策）

- 管理員驗證為客戶端常數比對，**不適合**高風險場景；若要強化，需引入 Firebase Auth 或後端驗證（見專案內討論紀錄／`01-專案總覽.md`）。
